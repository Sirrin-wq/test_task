При выполнении задачи было установлено, что предоставленная OpenAPI-спецификация не соответствует фактическому поведению API и содержит ошибки. Учитывая, что корректная документация является критически важным артефактом для разработки и интеграции, был выбран следующий стратегический подход:

1.  **Тестирование по спецификации как валидация требований**:
    Основной набор автотестов был написан **строго в соответствии с документацией**. Это позволяет явно **верифицировать контракт API** и автоматически детектировать любые отклонения реализации от формально задекларированных требований.

2.  **Фиксация расхождений как входных данных для разработки**:
    Все обнаруженные расхождения (как в логике, так и в структуре данных) были **задокументированы в виде отчета о несоответствиях** (см. `DISCREPANCIES.md`). Этот отчет выполняет две ключевые функции:
    *   Служит **формализованным списком дефектов документации** для немедленной правки.
    *   Выступает в роли **детального плана для рефакторинга API**, показывая, какие именно изменения необходимы для приведения реализации в соответствие со спецификацией.
